# 组件树与 Element 树分离设计及信号量方案

## 一、树分离：组件描述与渲染实例的彻底解耦

### 1. 当前耦合的痛点
在 Tenon 现有实现中，组件（如 `AntButton`）直接产出 Gio 的 `layout.Widget`，导致：
- **逻辑与渲染强绑定** – 修改属性必须重新执行整个渲染函数，无法局部更新。
- **状态通知复杂** – 因为没有独立的渲染节点，属性变更只能通过信号量、脏标记等跨层通知，增加了用户的心智负担。
- **难以复用持久化资源** – 每次更新都创建新 Widget，缺乏对底层图形对象的稳定持有。

### 2. 分离原则
将 UI 结构拆分为两层：
- **组件树 (Component Tree)** – 纯描述性结构，由开发者编写的轻量 Widget 构成。每次状态变化时，组件函数被重新执行，产生一棵**新的 Widget 树**。
- **Element 树 (Element Tree)** – 持久的、与屏幕节点一一对应的渲染实例。由框架自动管理，负责创建、更新、销毁真实的 Gio 渲染对象，并对外暴露局部更新方法。

### 3. 协作流程
1. **首次渲染**
    - 框架调用 `Component()` 获得 Widget 描述。
    - 框架将 Widget 实例化为 Element，创建对应的 Gio 底层节点并绘制。

2. **状态变更**
    - 信号或状态发生改变 → 关联的组件函数被标记为脏。
    - 下一帧，框架再次执行该组件函数，得到新的 Widget 树。
    - 框架将新 Widget 与旧 Element 进行**同级浅对比**（非全量 VDOM Diff）。
        - 若 Widget 类型相同，则复用 Element，仅更新变化的属性（如调用 `element.SetColor()`）。
        - 若类型不同或位置移动，则销毁旧 Element、创建新 Element。
    - 未变化的 Element 保持不动，避免不必要的重建。

### 4. 架构优势
- **对用户完全透明** – 用户只看见组件和普通方法调用（如 `f.SetColor("red")`），框架自动在 Element 层完成局部刷新。
- **极致更新性能** – 只更新发生变化的真实节点，零递归、零临时对象。
- **职责清晰** – 组件负责描述，Element 负责渲染与更新，各司其职。

## 二、信号量设计：类 React 19 的函数组件与 Hooks

### 1. 设计目标
提供接近 React 19 的组件开发体验：
- **纯函数组件** – UI 是状态的纯函数，每次渲染返回 Widget 描述。
- **Hooks 管理状态与副作用** – 通过 `useState`、`useEffect` 等 Hook 管理可变数据和生命周期。
- **自动依赖追踪** – 无需手动通知，Hook 内部自动订阅信号变化，精准触发重渲染。
